#### **C++ 编译**

总的来说，平衡编译时间和运行时性能是我们应该做的事。如果是内部链接，编译器能够做更多的优化，所以，如果一个实体仅仅用于单一编译单元，那么我们倾向于使用将其作为内部链接（如匿名类，匿名命名空间，static变量/函数等）。在头文件中，因为所有包含它的编译单元都会独立编译一次（因为头文件实际上就是在编译单元展开而已），所以我们希望头文件编译时间尽可能缩短，比如在我们已知所有模版可能的实例化情况下，我们可以通过在编译单元显式实例化来缩短编译时间，又比如使用pimpl来隐藏实现（但会导致额外的指针解引用开销以及内存不连续等影响性能的因素，需要trade-off）。同时，头文件不应该有内部链接属性，因为头文件并不指定编译单元，而我们前面提到内部链接的使用时强调了应该针对单一编译单元来使用，头文件使用内部链接的话所有包含其的编译单元都会产生一遍内部符号，导致二进制文件膨胀，这违背了头文件的初衷，所以在头文件中应避免匿名等内部链接属性。全局变量的定义我们总是只能放到编译单元来实现（为了符合ODR），但是现在有了inline，inline可以表示虽然每个编译单元都会编译一遍，但是最终只会保留一份符号，所以我们可以把很多实体放到头文件了，虽然放到头文件可以带来很多好处（比如避免循环依赖，减少编译单元，以及单编译文件内部进行编译器优化），但是inline不能避免对头文件多次重复的编译，这又是一个trade-off。在众多放到实体放到头文件的好处中，编译器优化是最重要的，幸运的是现在有了LTO可以在跨编译单元来实现这种优化，但可惜的是LTO也会增加编译时间，还是一样的trade-off。

- **前沿技术：ThinLTO。** LLVM/Clang 引入了 ThinLTO，GCC 也有类似的 WHOPR。ThinLTO 是一种可扩展的、多线程的 LTO 方案。它在编译阶段生成轻量级的摘要信息，在链接阶段只做全局的决策（比如谁可以被 inline），然后**并行**地将优化推回各个编译单元执行。**ThinLTO 极大地缓解了 LTO 的编译时间问题，是目前大型 C++ 项目（如 Chromium, LLVM 自身）的标配。**



#### **const，constexpr**

const 全局变量（非 inline）默认也是**内部链接**的，这意味着 const int MAX_SIZE = 100; 放在头文件里，每个包含它的 .cpp 都会生成一个独立的 MAX_SIZE。虽然通常会被优化成立即数，但如果取地址，就会产生多个不同的地址。**现代 C++ 推荐做法：在头文件中使用 inline constexpr 来定义全局常量。**

const 变量的值可以在运行期计算（例如 const int x = get_random();），而 constexpr 强制要求编译器在编译期计算出结果。这直接将运行时的计算开销转移到了编译期。

**消除存储与符号（Symbol）：** 因为 constexpr 的值在编译期完全确定，编译器通常会将其作为**立即数（Immediate Value）直接硬编码到机器码指令中。除非你对它进行了 odr-use（比如取了它的地址），否则它在目标文件（.o）中根本不占用数据段空间，也不产生链接符号**。

在 C++17 之前，如果你在类中声明了 static constexpr int x = 5;，一旦你对它进行了 odr-use（比如传给了一个接受 const int& 的函数），你**必须**在 .cpp 文件中提供一个类外定义（constexpr int MyClass::x;），否则链接器会报 undefined reference。
**C++17 规定，类的 static constexpr 数据成员隐式具有 inline 属性。** 这意味着你再也不需要写那个烦人的 .cpp 类外定义了！编译器和链接器会自动处理它的 ODR 问题，多个编译单元即使都 odr-use 了它，链接器也会自动合并，完美解决了历史遗留的链接痛点。

constexpr能够使用完整的编译期上下文。



#### **链接分类**

**外部链接 (External Linkage)** 

默认情况下，你在全局作用域或命名空间作用域声明的大多数东西，都是外部链接。

- **普通函数**：void foo() {}
- **全局变量**：int g_val = 0;
- **类/结构体/枚举**：class MyClass {};
- **命名空间**：namespace detail { void bar(); } （注意：detail 只是个名字，它依然是外部链接！）

**内部链接 (Internal Linkage)** 

内部链接的符号，只在当前 .cpp 文件内有效。链接器（Linker）根本看不到它们，或者把它们标记为局部符号。

- **static 修饰的全局变量/函数**：static int s_val = 0; static void helper() {}
- **全局 const 变量**（C++ 特有规则）：const int MAX_SIZE = 100;（除非你显式加了 extern）。
- **全局 constexpr 变量**。
- **匿名命名空间里的所有东西**。

**引入模版后的复杂情况 (Vague Linkage / 模糊链接)**

- **模版本身**：类模版和函数模版默认是**外部链接**。
- **模版实例化（COMDAT 折叠）**：如果 vector\<int> 在 10 个 .cpp 文件里被实例化了 10 次，它们都是外部链接。但链接器不会报 "multiple definition" 错误，而是会把这 10 份相同的汇编代码**折叠（Fold）成一份**。这在底层被称为 **Vague Linkage（模糊链接）**。
- **模版 + 内部链接的化学反应**：
  如果你用一个内部链接的类型（比如匿名命名空间里的类）去实例化一个外部链接的模版，**这个模版实例会被强制降级为内部链接**。

```C++
namespace { struct LocalTag {}; }
std::vector<LocalTag> v; // 这个 vector<LocalTag> 的实例只在当前 .cpp 可见！
```





#### **单一定义规则（One Definition Rule,  ODR）**

**什么是“使用”（odr-use）**

在C++中，“声明”告诉编译器名字和类型（比如 extern int x;），而“定义”则实际分配内存或提供代码实现（比如 int x = 0;）。

如果一个实体（变量、函数等）被**“使用”**了，程序中就必须有它的唯一**定义**。这里的“使用”在C++标准中有一个专门的术语叫 **odr-use**。

简单来说，如果编译器为了生成最终的机器码，**必须知道这个实体的内存地址，或者必须调用它的机器码**，那么这就构成了 odr-use。

- **显式使用：** 比如你读取一个变量的值、给变量赋值、调用一个函数、或者对变量取地址（&x）。
- **隐式使用：**
  - **new 表达式与 delete：** 当你 new 一个对象时，如果构造函数抛出异常，C++标准要求必须自动调用对应的 delete 来释放刚分配的内存。因此，即使你代码里没写 delete，编译器也会隐式地“使用”它，这就要求 delete 必须被定义。
  - **拷贝构造函数：** 即使编译器进行了优化（比如C++17的强制拷贝消除/Copy Elision），把拷贝操作优化没了，但在语义检查阶段，编译器依然认为拷贝构造函数被“使用”了，它必须存在且可访问。
  - **虚函数：** 只要一个类有虚函数（非纯虚函数），即使你代码里从来没有显式调用过它，它也被认为是“被使用了”。因为编译器需要把它的地址填入**虚函数表（vtable）**中，以支持多态。

**“不求值操作数”（Unevaluated Operand）**

C++中有几个特殊的运算符，比如 sizeof（求大小）和 decltype（推导类型）。
当你写 sizeof(x) 时，编译器**不需要去读取 x 所在的内存**，它只需要在编译期查一下符号表，知道 x 是个 int，直接就替换成 4 了。
这种不需要在运行期计算实际值的操作数，就叫**不求值操作数**。因为不需要实际访问，所以**不构成 odr-use，也就不要求 x 必须有定义**。

**typeid 的两面性**

- **情况 A：操作数是“非多态类型”（没有虚函数的类，或基本类型）**
  编译器在**编译期**就能确定它的类型。此时操作数是**不求值**的，不构成 odr-use。
- **情况 B：操作数是“多态类型”（包含至少一个虚函数的类）**
  因为存在多态（比如父类指针指向子类对象），编译器在编译期无法确定它的真实类型，必须在**运行期**去读取对象的内存（通常是通过虚函数表指针 vptr 找到 type_info）。这就必须访问该对象，因此**构成了 odr-use，要求该对象必须有定义**。

```C++
#include <typeinfo>

class Decider {
#if defined(DYNAMIC)
    virtual ~Decider() { // 如果定义了 DYNAMIC，这里就多了一个虚析构函数
    }
#endif
};

extern Decider d; // 仅仅是声明了 d，告诉编译器 d 在别的地方，但这里没有定义它！

int main() {
    char const* name = typeid(d).name(); // 关键点！
    return (int)sizeof(d);               // sizeof 永远是不求值的，没问题
}
```

- **假设没有定义宏 DYNAMIC：**
  Decider 就是一个空类，没有任何虚函数。它是**非多态类型**。
  typeid(d) 在编译期就能确定类型是 Decider。d 作为不求值操作数，**没有被 odr-use**。
  虽然 d 只有声明 extern Decider d; 没有定义，但因为没被真正“使用”，所以**程序合法，编译链接通过**。
- **假设定义了宏 DYNAMIC：**
  Decider 拥有了一个虚析构函数，它变成了**多态类型**。
  执行 typeid(d) 时，程序必须在运行期去内存里找 d 这个对象，看看它到底是个什么派生类（虽然这里没派生，但机制如此）。
  这就**构成了对 d 的 odr-use**。
  但是，代码里只有 extern Decider d;，整个程序中找不到 d 的定义（没有分配内存）。
  因此，**程序无效**。在实际操作中，编译器能通过，但**链接器（Linker）会报错**，提示 undefined reference to 'd'（未定义的引用）。

**头文件防卫式声明（Header Guards）的本质**

代码开头的 #ifndef GUARDDEMO_HPP 是我们非常熟悉的写法。这段文字指出了它的根本目的：**防止在同一个编译单元内违反 ODR**。
如果一个头文件定义了一个类（class X {}）或内联函数，而这个头文件被同一个 .cpp 文件直接或间接包含了两次，如果没有防卫声明，编译器就会看到两次 class X {}，这直接违反了“单一定义”规则，导致编译失败。

**什么时候必须有类的“完整定义”？**

核心逻辑是：**只要编译器需要知道这个类在内存中占多大空间，或者需要了解它的内部结构，就必须有完整定义。**

- **创建对象 / 作为成员变量：** 编译器得分配内存，必须知道大小。
- **使用 sizeof 或 typeid：** 必须知道大小和多态信息。
- **访问成员：** 必须知道类里面有什么。
- **类型转换（static_cast / dynamic_cast）：** 编译器需要计算指针偏移量（特别是多重继承时），必须知道类的继承结构。
- *(反例：如果只是声明一个指针 X\* ptr; 或作为函数声明的参数 void func(X);，则不需要完整定义，因为指针大小是固定的，声明函数也不产生实际调用代码。)*



#### **跨编译单元的等价性约束**

如果同一个实体（比如一个内联函数或类）在多个编译单元中被定义，C++ 标准要求它们必须**完全等价**。如果违反了，属于**未定义行为（Undefined Behavior, UB）**，而且标准**不要求编译器报错（NDR）**。这意味着程序可能崩溃，可能逻辑错误，也可能“碰巧”正常运行，极其难以排查。

“完全等价”包含两个极其严格的条件：

**条件一：词法标记序列完全一致（长得一模一样）**

两个定义中的关键字、变量名、符号等，在经过预处理（宏展开）后，必须一个字都不差。

**条件二：上下文含义必须一致（灵魂也得一模一样）**

这是最容易踩坑的地方。**哪怕代码长得一模一样，如果它们引用的外部实体不同，依然违反 ODR！**

```C++
// 编译单元 1 和 2 中都有完全相同的代码：
static int counter = 0;
inline void increaseCounter() { ++counter; }
```

- **表象：** 两个 increaseCounter 函数的代码文本完全一样。
- **真相：** static int counter 在全局作用域意味着**内部链接（Internal Linkage）**。也就是说，编译单元 1 有一个属于自己的 counter，编译单元 2 也有一个属于自己的、物理地址完全不同的 counter。
- **灾难：** increaseCounter 是内联函数，C++ 规定整个程序中所有的 increaseCounter 必须是同一个实体。但是，单元 1 的函数操作的是单元 1 的 counter，单元 2 的函数操作的是单元 2 的 counter。**相同的标记 counter 指代了不同的实体！** 这严重违反了 ODR。

- **Example 2：** 单元 1 声明 void unused(int = 3);，单元 2 声明 void unused(int = 4);。默认参数不同，直接违反 ODR。

- **Example 3：**

```C++
//编译单元1
class X {
    public:
        X(int, int);
        X(int, int, int);
};

X::X(int, int = 0)
{
}

class D {
    X x = 0;
};

D d1; //X(int, int)被D()调用


//编译单元2
class X {
    public:
        X(int, int);
        X(int, int, int);
};

X::X(int, int = 0, int = 0)
{
}

class D : public X {
    X x = 0;
}

D d2; //X(int, int, int)被D()调用
```

- 单元 1 中，类 X 的构造函数默认参数导致 D d1; 隐式调用了 X(int, int)。
- 单元 2 中，类 X 的构造函数默认参数导致 D d2; 隐式调用了 X(int, int, int)。
- **结果：** 编译器为类 D 隐式生成的默认构造函数 D::D()，在两个编译单元里生成了**不同的机器码**（一个调 2 参函数，一个调 3 参函数）。同一个类 D 的默认构造函数在不同文件里行为不一致，违反 ODR！



#### **规则的例外（C++ 的实用主义妥协）**

```C++
// header.hpp
int const length = 10;
class MiniBuffer { char buf[length]; };
```

当这个头文件被多个 .cpp 包含时：

1. 在 C++ 中，全局的 const 变量默认是**内部链接**（相当于隐式加了 static）。
2. 因此，每个 .cpp 文件都会生成一个自己专属的 length 变量。
3. 按照前面的严格规则，MiniBuffer 类中的 length 标记在不同的 .cpp 中指代了不同的物理变量，应该违反 ODR 才对！

因为 C++ 标准规定：如果完全一样的标记指代了不同的实体，但这些实体是 **const 常数**，且它们具有**相同的值**，并且在代码中**没有使用它们的地址**（只是读取它们的值，比如用来定义数组大小），那么**不视为违反 ODR**。

这是一个非常务实的例外。如果不允许这样做，我们就无法在头文件中使用 const 变量来定义数组大小了（在 C++11 引入 constexpr 之前，这是唯一的做法）。
